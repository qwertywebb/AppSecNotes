# Отчёт о прохождении машины Silent Monitor

## 1. Разведка (Reconnaissance)

### 1.1 Nmap сканирование

Выполнено сканирование Nmap для выявления открытых портов и определения сервисов целевой системы.

```bash
nmap -sV -sC -p- 10.113.175.93
```

Результаты сканирования показали следующие открытые порты:

- **22/tcp** — SSH (OpenSSH)
- **5050/tcp** — Werkzeug HTTP сервер 2.0.2 (Python 3.10.12)

На порту 5050 обнаружен веб-сервер, на котором работает NOC Portal версии 2.4.1.


## 2. Веб-разведка и анализ приложения

### 2.1 Фаззинг веб-директорий

Проведён фаззинг для выявления скрытых директорий и эндпоинтов.

```bash
ffuf -w /usr/share/wordlists/dirb/common.txt -u http://10.113.175.93:5050/FUZZ
```

Обнаружена директория `/internal` (статус 200, размер 8770 байт). При попытке доступа к `/internal` отображается форма входа.

При переборе путей внутри `/internal` найдены следующие эндпоинты:

- `/internal/health` — статус 200, размер 8770 байт (требует авторизации)
- `/internal/logout` — статус 200, размер 8770 байт (требует авторизации)
- `/internal/dashboard` — статус 200, размер 8770 байт (требует авторизации)

Доступ к этим эндпоинтам без авторизации невозможен — происходит перенаправление на страницу входа.

### 2.2 Анализ версии Werkzeug

Версия `Werkzeug/2.0.2` была проверена на наличие известных уязвимостей, включая доступ к дебаг-консоли (`/console`). Путь `/console` не обнаружен. Попытки эксплуатации стандартных уязвимостей Werkzeug результата не дали.

### 2.3 Перебор виртуальных хостов (vhosts)

Проведён перебор виртуальных хостов с целью обнаружения скрытых поддоменов или внутренних сервисов. Результатов не получено.


## 3. Анализ проксирования запросов

При анализе заголовков HTTP-ответа обнаружено, что сервер Werkzeug на порту 5050 проксирует запросы на внутренний адрес `127.0.0.1:8080`.

**Параметры ответа:**

- Схема: http
- Хост: 10.113.175.93:5050
- Имя файла: /internal
- Адрес: 127.0.0.1:8080
- Статус: 200 OK
- Версия: HTTP/1
- Сервер: Werkzeug/2.0.2 Python/3.10.12

Это указывает на то, что внешний сервер выступает в роли прокси для внутреннего приложения, работающего на `localhost:8080`. Такая архитектура может скрывать уязвимости внутреннего сервиса, которые не доступны напрямую извне.


## 4. SQL-инъекция в форме входа

### 4.1 Обнаружение уязвимости

В форме логина на странице `/internal` проведено тестирование параметров `username` и `password`. Установлено, что параметр `username` уязвим для SQL-инъекции.

**Проверочный пейлоад:**

```
username=1' OR 1=1 -- &password=test
```

После отправки запроса сервер вернул cookie сессии, что позволило получить доступ к `/internal/dashboard`.

### 4.2 Получение доступа к панели управления

Используя полученную cookie, был осуществлён вход в систему и получен доступ к панели управления NOC.


## 5. Command Injection в /internal/health

### 5.1 Обнаружение уязвимости

Внутри панели управления обнаружен сервис `/internal/health`, который предназначен для проверки доступности хостов через команду `ping`. Параметр `target` подставляется в системную команду без достаточной фильтрации.

Фильтр блокирует символы `; | & $` и пробелы, но не блокирует символ новой строки (`%0a`). Это позволяет выполнить произвольные команды через URL-кодирование.

**Успешный пейлоад:**

```
127.0.0.1%0awhoami
```

**Результат выполнения:**

```
www-data
```

Команда выполнилась от пользователя `www-data`. Уязвимость Command Injection подтверждена.

### 5.2 Чтение конфигурационного файла

Через Command Injection был прочитан файл `secret.config`, содержащий учётные данные.

**Пейлоад:**

```
192.168.130.144%0acat secret.config
```

**Содержимое файла:**

```
run_as   = sysadmin
password = S3cur3Backup$Acc3ss!
```

### 5.3 Получение доступа по SSH

Используя найденные учётные данные (`sysadmin:S3cur3Backup$Acc3ss!`), выполнен вход на целевой сервер по SSH.

```bash
ssh sysadmin@10.113.175.93
```

В домашней директории пользователя `sysadmin` найден файл `user.txt` с флагом пользователя.


## 6. Пост-эксплуатация и обнаружение базы данных KeePass

### 6.1 Обнаружение базы данных

В директории `/home/sysadmin/backups` обнаружен файл `infrastructure.kdbx` — база данных KeePass, которая может содержать пароли от инфраструктурных сервисов, включая учётную запись `root`.

Файл был скопирован на атакующую машину для последующего анализа и взлома.

### 6.2 Попытки взлома KDBX 4

**Первая попытка — keepass2john + hashcat**

Стандартный `keepass2john` из состава John the Ripper (версия 1.9.0-Jumbo-1) не поддерживает формат KDBX 4 (версия файла `40000`). Ошибка:

```
! infrastructure.kdbx : File version '40000' is currently not supported!
```

**Вторая попытка — keepass4brute**

Утилита `keepass4brute` была запущена со словарём `rockyou.txt`. Процесс перебора занял значительное время и не принёс результата (пароль не найден).

**Третья попытка — KDBXcrack**

Установлен и запущен скрипт `KDBXcrack`. Аналогично, пароль из стандартного словаря подобран не был.

### 6.3 Поиск пароля в исходном коде приложения

При анализе исходного кода веб-приложения (найденного через чтение файлов через Command Injection) обнаружен фрагмент инициализации базы данных:

```python
pw = hashlib.md5(b"C0rp$N3tS3cur3!").hexdigest()
```

Пароль `C0rp$N3tS3cur3!` был проверён как мастер-пароль KeePass. Не подошёл.

### 6.4 Обнаружение SSH-ключа

В директории `/var/lib/fwupd/pki/` найден приватный ключ `secret.key`. Ключ был проверён для подключения к `root`, но не подошёл — публичная часть не добавлена в `authorized_keys` для пользователя `root`.

### 6.5 Запуск LinPEAS

На целевой системе был запущен скрипт `linpeas.sh` для автоматизированного поиска векторов эскалации привилегий. Ничего критического не обнаружено.


## 7. Успешный взлом базы данных KeePass

### 7.1 Повторный запуск keepass4brute с терпением

После того как стандартные словари не дали результата, процесс `keepass4brute` был оставлен работать на продолжительное время. В процессе перебора был найден пароль.

**Вывод утилиты:**

```
[+] Words tested: 2526/14344393 - Attempts per minute: 886 - Estimated time remaining: 1 weeks, 4 days
[+] Current attempt: spring

[*] Password found: spring
```

Мастер-пароль базы данных KeePass оказался простым словом `spring`.

### 7.2 Извлечение пароля root из базы данных

После успешного открытия базы данных `infrastructure.kdbx` (через `keepassxc` или `keepassxc-cli`) была найдена запись, содержащая пароль для пользователя `root`.

**Найденные учётные данные:**

```
root: S3cur3P4ss0nK33p4ss
```

### 7.3 Переключение на root и получение флага

Используя пароль `S3cur3P4ss0nK33p4ss`, выполнен вход на целевой сервер как `root`.

```bash
su root
Password: S3cur3P4ss0nK33p4ss
```

В домашней директории `root` найден файл `root.txt` с финальным флагом.


## 8. Итог

### 8.1 Цепочка атаки

1. **Разведка:** Nmap сканирование, обнаружение портов 22 (SSH) и 5050 (Werkzeug HTTP).
2. **Фаззинг:** Обнаружение директории `/internal` и эндпоинтов `/health`, `/dashboard`, `/logout`.
3. **Анализ проксирования:** Выявлено, что Werkzeug проксирует запросы на `127.0.0.1:8080`.
4. **SQL-инъекция:** Обход аутентификации через пейлоад `1' OR 1=1 --` в поле `username`.
5. **Command Injection:** Эксплуатация уязвимости в `/internal/health` через `%0a` (новая строка), чтение файла `secret.config`.
6. **Получение доступа по SSH:** Использование учётных данных `sysadmin:S3cur3Backup$Acc3ss!`.
7. **Обнаружение базы данных KeePass:** Файл `infrastructure.kdbx` скопирован на атакующую машину.
8. **Взлом KeePass:** Подбор мастер-пароля `spring` через `keepass4brute`.
9. **Извлечение пароля root:** Из базы данных получен пароль `S3cur3P4ss0nK33p4ss`.
10. **Переключение на root:** Получен финальный флаг.

### 8.2 Найденные учётные данные

| Пользователь | Пароль | Источник |
|--------------|--------|----------|
| `sysadmin` | `S3cur3Backup$Acc3ss!` | `secret.config` |
| `root` | `S3cur3P4ss0nK33p4ss` | `infrastructure.kdbx` (KeePass) |
| Мастер-пароль KeePass | `spring` | Подбор через `keepass4brute` |

### 8.3 Достижения и новые навыки

**SQL-инъекция в форме входа** — отработана техника обхода аутентификации через классический пейлоад `1' OR 1=1 --`. Несмотря на наличие фильтрации некоторых ключевых слов, простая инъекция сработала.

**Command Injection через URL-кодирование (`%0a`)** — впервые применена техника обхода фильтрации символов через подстановку новой строки (`%0a`). Позволила выполнить произвольные команды от пользователя `www-data`.

**Анализ проксирования запросов** — обнаружено, что внешний сервер проксирует трафик на `localhost:8080`. Понимание архитектуры помогло в дальнейшем анализе уязвимостей.

**Работа с KeePass (KDBX 4)** — освоены методы взлома баз данных нового формата. Изучена проблема несовместимости стандартного `keepass2john` с KDBX 4 и альтернативные инструменты (`keepass4brute`, `KDBXcrack`). Получен опыт длительного брутфорса с большими словарями.

**Чтение файлов через Command Injection** — отработана техника чтения конфиденциальных файлов через команду `cat` в параметре `target`.

**SSH доступ с найденными учётными данными** — получен доступ к системе с правами пользователя `sysadmin`, что позволило продолжить исследование и найти базу данных KeePass.

**Взлом KeePass с нетривиальным паролем** — мастер-пароль `spring` был успешно подобран через `keepass4brute` после длительного ожидания. Пароль оказался простым словом из английского языка.

**Получение root-доступа** — финальная эскалация через пароль `root`, извлечённый из базы данных KeePass.
