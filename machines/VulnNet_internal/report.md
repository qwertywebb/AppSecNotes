# Отчёт о прохождении VulnNet Internal

## 1. Разведка (Reconnaissance)

Сканирование Nmap выявило открытые порты: 22 (SSH), 111 (rpcbind), 139 и 445 (SMB), 6379 (Redis), 9090 (zeus-admin), 33275 (Java RMI). Службы SMB привлекли внимание прежде всего, так как часто содержат ценные данные.

## 2. SMB — первый вектор проникновения

С помощью smbclient были обнаружены доступные шары: `shares`, `print$`, `IPC$`. Шара `shares` оказалась доступна без пароля. Внутри лежали три файла: `services.txt`, `data.txt`, `business-req.txt`. В одном из них находился первый флаг.

## 3. Java RMI — попытка эксплуатации

Порт 33275 был идентифицирован как Java RMI. Эксплойт Metasploit (multi/misc/java_rmi_server) не сработал: сервер сообщил, что RMI class loader отключён. Вектор закрыт.

## 4. NFS — находка конфигурации Redis

Через `showmount -e` была обнаружена NFS-шара `/opt/conf`, доступная для монтирования любому клиенту. После монтирования в локальную директорию в ней был найден файл `redis.conf`, содержащий пароль для доступа к Redis: `B65Hx562F@ggAZ@F`. Без этого пароля дальнейшая работа с Redis была бы невозможна, так как подключение без пароля отклонялось.

## 5. Redis — получение данных и нового пароля

С помощью найденного пароля была установлена аутентификация в Redis. Команда `KEYS *` показала ключи: `marketlist`, `int`, `internal flag`, `authlist`, `tmp`. Команда `GET "internal flag"` вернула флаг.

В ключе `authlist` хранилась закодированная строка, которая после декодирования раскрыла пароль для rsync: `Hcg3HP67@TW@Bc72v`. Так был найден пароль для следующего сервиса.

## 6. Rsync — получение доступа к файлам

С помощью пароля `Hcg3HP67@TW@Bc72v` был получен доступ к модулю rsync. Через него был скачан файл `user.txt` (флаг). SSH-ключи пользователей ubuntu и sys-internal отсутствовали, поэтому был сгенерирован свой ключ и загружен на сервер через rsync.

## 7. SSH — вход в систему

Сгенерированный SSH-ключ позволил подключиться к серверу под пользователем sys-internal и получить стабильную оболочку.

## 8. TeamCity — эскалация привилегий

На сервере обнаружен TeamCity (CI/CD), работающий локально на порту 8111. Через SSH-туннель порт был проброшен на локальную машину.

TeamCity версии 2020.2.2 оказался уязвим к CVE-2024-56348. Эксплойт создал пользователя с правами администратора, но не смог автоматически сгенерировать токен. Пользователь был создан, и через веб-интерфейс был создан токен вручную.

Через API TeamCity была создана build configuration с шагом Command Line. В качестве команды был использован reverse shell на Python (так как стандартный bash не работал в окружении агента). Агент сборки запустил команду от имени root, так как TeamCity работал с максимальными привилегиями.

Reverse shell успешно пришёл на слушатель, и была получена root-оболочка.

## 9. Итог

Цепочка атаки включала: SMB → NFS (redis.conf) → Redis → Rsync → SSH → TeamCity (CVE-2024-56348) → reverse shell → root. Ключевыми моментами стали извлечение пароля Redis из NFS-шары, получение пароля Rsync из Redis, и эскалация прав через уязвимую версию TeamCity.

## 10. Достижения и новые навыки

В ходе прохождения машины VulnNet Internal были впервые применены следующие техники и инструменты:

**Rsync** — ранее не использовался в практике. Удалось не только получить доступ к файлам через анонимный модуль, но и загрузить SSH-ключ для получения постоянного доступа. Понимание работы rsync как вектора для передачи файлов и закрепления в системе теперь отражено в арсенале.

**Эксплуатация CI/CD (TeamCity)** — впервые проведена атака на систему непрерывной интеграции. Изучена уязвимость CVE-2024-56348, позволяющая создавать административных пользователей без аутентификации. Освоен процесс создания build configuration, добавления шага с командой и запуска сборки для выполнения произвольного кода на сервере. Получен опыт обхода ограничений, когда стандартный reverse shell на bash не работал из-за особенностей окружения агента (использован Python reverse shell).

**Redis с аутентификацией** — до этого момента встречался только Redis без пароля. Впервые был найден пароль в конфигурационном файле через NFS, и выполнена аутентификация для извлечения данных.

**NFS** — впервые смонтирована удалённая NFS-шара и использована для поиска конфиденциальных файлов (redis.conf).

**Java RMI** — несмотря на неудачную эксплуатацию, получен опыт идентификации сервиса и понимания ограничений (отключённый class loader).
