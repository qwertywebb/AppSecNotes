# KERBEROS — ПОЛНЫЙ РАЗБОР

Kerberos — протокол аутентификации, использующий систему билетов (tickets).
Назван в честь Цербера — трехголового пса, охраняющего врата ада.

КЛЮЧЕВОЕ ОТЛИЧИЕ ОТ NETNTLM:
- NetNTLM: клиент → сервис → DC (сервис пересылает)
- Kerberos: клиент → DC → получает билет → предъявляет сервису (сервис проверяет сам)

## Компоненты Kerberos

- KDC (Key Distribution Center) — служба на DC, которая выдает билеты.
  Состоит из двух частей:
  * AS (Authentication Service) — выдает TGT
  * TGS (Ticket Granting Service) — выдает TGS (билеты на сервисы)

- TGT (Ticket Granting Ticket) — "первичный билет". 
  Выдается после успешной аутентификации. Позволяет запрашивать билеты на сервисы.
  Зашифрован хэшом учетной записи KRBTGT.

- TGS (Ticket Granting Service) / Service Ticket — билет на конкретный сервис.
  Зашифрован хэшом аккаунта, под которым работает сервис.

- SPN (Service Principal Name) — уникальный идентификатор сервиса в AD.
  Формат: тип_сервиса/имя_хоста (пример: cifs/fileserver.thm.local)

- KRBTGT — специальная учетная запись в AD.
  Ее хэш используется для шифрования всех TGT.
  КОМПРОМИСС KRBTGT = ПОЛНЫЙ КОНТРОЛЬ НАД ДОМЕНОМ (Golden Ticket)

## Как работает Kerberos (5 шагов)

ШАГ 1: AS-REQ (Authentication Service Request)
Клиент → KDC: отправляет username и timestamp, зашифрованный хэшом пароля пользователя.
Это называется pre-authentication.

ШАГ 2: AS-REP (Authentication Service Response)
KDC: расшифровывает timestamp хэшом пользователя (из AD).
Если успешно → KDC отправляет клиенту:
- Session Key (зашифрован хэшом пользователя)
- TGT (зашифрован хэшом KRBTGT)

Клиент расшифровывает Session Key, но не может расшифровать TGT.

ШАГ 3: TGS-REQ (Ticket Granting Service Request)
Клиент хочет получить доступ к сервису (например, файловому серверу).
Клиент → KDC: отправляет TGT + SPN сервиса + authenticator (username + timestamp, зашифрованный Session Key)

ШАГ 4: TGS-REP (Ticket Granting Service Response)
KDC: расшифровывает TGT хэшом KRBTGT, проверяет SPN.
Отправляет клиенту:
- Service Ticket (TGS) — зашифрован хэшом аккаунта сервиса
- Service Session Key — зашифрован Session Key клиента

ШАГ 5: AP-REQ (Application Request)
Клиент → Сервис: отправляет Service Ticket
Сервис: расшифровывает Service Ticket своим хэшом, проверяет, пускает клиента.

## Преимущества Kerberos

- Взаимная аутентификация (клиент проверяет сервис, сервис проверяет клиента)
- Пароль/хэш никогда не передается по сети (только билеты)
- Single Sign-On (SSO) — один раз ввел пароль, получил TGT на 10 часов
- Сервис может сам проверить билет (не обращаясь к DC)
- Билеты имеют ограниченное время жизни

## Недостатки Kerberos (для пентестера — возможности!)

- Требует синхронизации времени (расхождение >5 минут → ошибка)
- KDC — единая точка отказа
- Билеты можно украсть и использовать (Pass-the-Ticket)
- KRBTGT хэш можно украсть → Golden Ticket (подделка TGT для любого пользователя)
- Kerberoasting: любой пользователь может запросить TGS для SPN и взломать офлайн

## Kerberos с Impacket (практика)

# Добавить хост в /etc/hosts (Kerberos работает через hostname, не IP!)
echo "192.168.11.51 SERVER1.thm.loc" >> /etc/hosts

# Получить TGT
getTGT.py thm.loc/mary:'SuperLongForKerberos123!' -dc-ip 192.168.11.100

# Будет создан файл mary.ccache

# Использовать билет для аутентификации (3 способа)
# KRB5CCNAME - Linux переменная, которая указывает на расположение файла кэша кредов 
export KRB5CCNAME=mary.ccache
smbclient.py thm.loc/mary@SERVER1.thm.loc -k -no-pass -dc-ip 192.168.11.100

# Или без export (одной строкой)
KRB5CCNAME=mary.ccache smbclient.py thm.loc/mary@SERVER1.thm.loc -k -no-pass -dc-ip 192.168.11.100

# Или через флаг -ccache (в новых версиях)
smbclient.py thm.loc/mary@SERVER1.thm.loc -k -no-pass -dc-ip 192.168.11.100 -ccache mary.ccache

## ccache файлы

ccache — файл кэша билетов Kerberos в Linux.

Стандартное расположение: /tmp/krb5cc_%{uid}
Переменная окружения: KRB5CCNAME — указывает какой файл использовать.

Посмотреть билеты в кэше:
klist

# ЧЕТЫРЕ ОСНОВНЫХ АТАКИ НА KERBEROS

1. Golden Ticket
   - Подделка TGT с использованием хэша KRBTGT
   - Результат: полный контроль над доменом, доступ к любым ресурсам
   - Требует: хэш KRBTGT (нужны права на DC для получения)

2. Silver Ticket
   - Подделка TGS для конкретного сервиса с использованием хэша этого сервиса
   - Результат: доступ к конкретному серверу/сервису
   - Требует: хэш аккаунта сервиса (например, компьютера)

3. Pass-the-Ticket (PtT)
   - Использование украденного TGT или TGS без знания пароля
   - Результат: выдать себя за пользователя
   - Требует: доступ к кэшу билетов (ccache или память lsass)

4. Kerberoasting
   - Запрос TGS для SPN и взлом офлайн
   - Результат: пароль сервисной учетки (часто с правами)
   - Требует: любой доменный пользователь

# КЛЮЧЕВЫЕ МЫСЛИ ПО KERBEROS

1. TGT нужен для ЗАПРОСА доступа к сервисам. Сам по себе он не дает доступ.
2. Kerberos требует hostname, потому что SPN привязан к DNS имени.
3. Без синхронизации времени ±5 минут Kerberos не работает.
4. TGS можно запросить у KDC, имея TGT и права пользователя на этот сервис.
