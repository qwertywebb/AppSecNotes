# NETNTLM — ПОЛНЫЙ РАЗБОР

NetNTLM — это challenge-response протокол. Пароль не передается по сети, только ответ на вызов (challenge).

## Версии NetNTLM:

- NTLMv1 — старая, очень небезопасная
- NTLMv2 — современная, но все еще уязвимая

## Как работает NetNTLM (по шагам):

1. Клиент → Сервер: "хочу доступ, мой username"
2. Сервер → Клиент: случайное 16-байтное число (challenge)
3. Клиент: берет NT хэш своего пароля, шифрует им challenge, отправляет response
4. Сервер → Domain Controller: пересылает username, challenge, response
5. DC: находит NT хэш пользователя в своей базе, шифрует им тот же challenge
6. DC: сравнивает свой результат с response от клиента
7. DC → Сервер: "да" или "нет"
8. Сервер → Клиент: доступ разрешен или запрещен

## Преимущества NetNTLM (почему еще жив):

- Простой в реализации
- Не требует синхронизации времени (в отличие от Kerberos)
- Работает без домена (в Workgroup)
- Fallback когда Kerberos недоступен

## Недостатки NetNTLM (почему это интересно пентестеру):

- Нет взаимной аутентификации — клиент не проверяет сервер (можно MITM)
- NTLMv1 — слабая криптография
- Уязвим для relay-атак (перехваченный response можно отправить на другой сервер)
- Уязвим для Pass-the-Hash (зная хэш, можно авторизоваться без пароля)
- Каждый запрос требует обращения к DC (медленнее чем Kerberos)

# SPN (Service Principal Name)

SPN = идентификатор сервиса в AD. Формат: тип_сервиса/имя_хоста.

Kerberos НЕ работает без SPN. Если SPN нет — клиент использует NetNTLM.

Для пентестера:
- SPN = цель для Kerberoasting (запрос TGS и взлом офлайн)
- Отсутствие SPN = принуждение к NetNTLM (можно ретранслировать)
- Атака "принудительная аутентификация" (coerced auth) заставляет сервер инициировать NetNTLM к атакующему

## Когда используется NetNTLM (даже если Kerberos по умолчанию):

- Клиент не может связаться с DC
- Доступ по IP-адресу (вместо hostname)
- У сервиса нет зарегистрированного SPN в AD
- Подключение к машине не в домене
- Legacy приложения

## Практика с Impacket smbclient.py:

Команда для подключения к SMB шаре через NetNTLM:
smbclient.py домен/пользователь:'пароль'@IP

Пример:
smbclient.py thm.loc/claire:'Password123!'@192.168.11.51

После подключения:
- shares — посмотреть доступные шары
- use SHARE1 — подключиться к конкретной шаре

# NETNTLM ДЛЯ ПЕНТЕСТА — КЛЮЧЕВЫЕ ВЕКТОРЫ

1. Pass-the-Hash (PtH):
   - Зная NT хэш пользователя, можно авторизоваться без пароля
   - Работает потому что NetNTLM использует хэш напрямую

2. NTLM Relay:
   - Перехваченный NetNTLM response можно ретранслировать на другой сервер
   - Пример: перехватил response от пользователя на HTTP → отправил на SMB
   - Можно получить доступ к файловому серверу

3. NTLMv1 уязвимость:
   - Хэши NTLMv1 можно разломать (сollect, offline crack)
   - Используй если видишь NTLMv1 в сети

4. Без SPN = NetNTLM:
   - Если сервис не имеет SPN, Kerberos не работает
   - Атакующий может принудить клиента использовать NetNTLM (специфические атаки)
   

# NetNTLM — получение пароля из response

Перехваченный NetNTLMv2 response НЕЛЬЗЯ использовать для Pass-the-Hash.
Его нужно взламывать офлайн:

1. Сохранить в формате: username::domain:challenge:response
2. Hashcat: -m 5600 (NetNTLMv2)
3. John: --format=netntlmv2

Результат — обычный пароль пользователя.
