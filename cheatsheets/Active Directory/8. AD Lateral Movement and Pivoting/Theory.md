# LATERAL MOVEMENT & PIVOTING — ДВИЖЕНИЕ ПО СЕТИ

## Что это

Lateral movement — это перемещение от одного скомпрометированного хоста к другому с использованием легитимных учетных данных (пароли, хэши, билеты).

В отличие от начального взлома, мы НЕ эксплуатируем уязвимости. Мы используем украденные ключи и ходим через легитимные протоколы администрирования.

---

## Зачем это нужно

Один скомпрометированный компьютер редко содержит что-то ценное.

Ценные данные находятся:
- Domain Controller
- Серверы с правами
- Backup серверы

Чтобы до них добраться, нужны учетные данные с этих машин.

---

# ТРИ ОСНОВНЫЕ ТЕХНИКИ LATERAL MOVEMENT

## 1. Remote Execution (удаленное выполнение)

Запуск команд на удаленной машине через легитимные протоколы администрирования Windows.

Инструменты:
- PsExec (через SMB)
- WinRM (интерактивная сессия через HTTP/HTTPS)
- WMI
- DCOM

## 2. Credential Reuse (повторное использование учеток)

Аутентификация с использованием украденных хэшей или билетов вместо пароля.

Техники:
- Pass-the-Hash (NTLM хэш)
- Pass-the-Ticket (Kerberos билеты)
- Overpass-the-Hash (хэш → билет)

## 3. Pivoting (переключение через промежуточный хост)

Использование скомпрометированного хоста как реле для доступа к сетям, которые напрямую недоступны.


# ЧТО НУЖНО ДЛЯ LATERAL MOVEMENT

## Два обязательных условия:

1. **Валидные учетные данные**
   - Plaintext пароль
   - NTLM хэш
   - Kerberos билет

2. **Административный доступ на цели**
   - Учетка должна быть в группе **Local Administrators** на целевой машине
   - Или в **Remote Management Users** (для WinRM)

## Важный нюанс

Local Admin ≠ Domain Admin.
Пользователь может быть локальным админом на нескольких рабочих станциях, но не иметь прав в домене.

НО: если на одной из этих станций есть активная сессия Domain Admin → можно украсть его креды.


# ИНСТРУМЕНТЫ ДЛЯ LATERAL MOVEMENT

## Impacket (Python)

- `psexec.py` — запуск команд через SMB (создает службу)
- `wmiexec.py` — через WMI (тише, нет служб)
- `smbexec.py` — через SMB (альтернатива)
- `dcomexec.py` — через DCOM
- `atexec.py` — через планировщик задач (Scheduled Tasks)

## NetExec (nxc)

- Мультипротокольный инструмент
- Поддерживает SMB, WinRM, RDP, LDAP, MSSQL

## Evil-WinRM

- Интерактивные WinRM сессии
- Поддержка загрузки/скачивания файлов
- Загрузка скриптов в память

## Для Pivoting

- Chisel
- Ligolo-ng


# КЛЮЧЕВОЙ ЦИКЛ LATERAL MOVEMENT

```
Переместиться на новый хост
      │
      ▼
Собрать новые учетные данные (LSASS, SAM, кэш)
      │
      ▼
Переместиться дальше
      │
      ▼
Повторять до Domain Admin
```


# ВАЖНЫЕ НЮАНСЫ

- Большинство техник требует **Local Admin** на цели
- Pass-the-Hash работает только с NTLM (не с Kerberos)
- Pass-the-Ticket требует импорта билета в текущую сессию
- Pivoting нужен когда цель в другой сети (VLAN, Management network)


# БЫСТРАЯ ШПАРГАЛКА

# Проверить доступ через SMB с паролем
nxc smb TARGET_IP -u username -p password

# Pass-the-Hash через SMB
nxc smb TARGET_IP -u username -H :NTLM_HASH

# Получить shell через psexec
psexec.py DOMAIN/user@IP -hashes :NTLM_HASH

# Интерактивная WinRM сессия
evil-winrm -i TARGET_IP -u username -p password

# WinRM с хэшем
evil-winrm -i TARGET_IP -u username -H NTLM_HASH

# Запустить команду через WMI
wmiexec.py DOMAIN/user@IP -hashes :NTLM_HASH "whoami"

# УТОЧНЕНИЯ ПО LATERAL MOVEMENT

## Два обязательных условия:
1. Валидные учетные данные (пароль, хэш, билет)
2. Учетка должна быть в группе **Local Administrators** на цели (или Remote Management Users для WinRM)

Не "админская учетка", а "учетка с локальными админскими правами на этой конкретной машине".

## Инструменты Impacket:
- psexec.py — создает службу (шумный) через админскую шару
- wmiexec.py — через WMI (тише)
- smbexec.py — альтернативный SMB
- atexec.py — через планировщик задач

## Evil-WinRM
- Использует протокол WinRM (порты 5985/5986)
- Для работы нужна группа Remote Management Users (или локальный админ)

# LOCAL ADMIN VS SYSTEM

SYSTEM — это максимальные привилегии на ОДНОМ компьютере.

Для локальных операций:
- Большинство команд Mimikatz требуют Admin (SYSTEM работает лучше)
- lsadump::cache и lsadump::sam требуют SYSTEM

Для удаленных операций (lateral movement):
- SYSTEM на текущей машине НЕ дает прав на другие машины
- Нужны креды учетки с Local Admin правами на цели

Вывод: SYSTEM полезен для дампа credentials с текущей машины.
Для движения дальше нужны сами креды, а не уровень SYSTEM.

# ЧТО НУЖНО ДЛЯ ПЕРЕХОДА НА ДРУГУЮ МАШИНУ

Текущий уровень привилегий на машине A (Local Admin/SYSTEM) НЕ ДАЕТ прав на машину B.

Для входа на машину B нужны:
- Пароль или NTLM хэш учетной записи
- Эта учетная запись должна иметь Local Admin права на машине B

Откуда взять такие креды:
- Дамп LSASS на A (если админ логинился на A)
- DCC2 кэш на A → взлом → пароль
- Kerberos билеты (если есть TGT или TGS)
- Другие источники (файлы, базы, скрипты)

Типичный путь:
WRK (SYSTEM) → дамп LSASS → нашли хэш Domain Admin → подключаемся к DC
