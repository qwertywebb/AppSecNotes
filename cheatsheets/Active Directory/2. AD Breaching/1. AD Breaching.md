# BREACHING AD — ПОЛУЧЕНИЕ ПЕРВЫХ УЧЕТНЫХ ДАННЫХ

## Что это такое

Breaching — это процесс получения первых валидных учетных данных в домене, когда у тебя ничего нет.
Это САМЫЙ ПЕРВЫЙ этап атаки на AD.

Без первого аккаунта ты не можешь:
- Перечислять объекты в домене
- Двигаться по сети
- Повышать привилегии

## Почему даже низкопривилегированная учетка ценна

Даже обычный доменный пользователь (без прав) дает возможность:
- Запросить список всех пользователей, групп, компьютеров
- Посмотреть групповые политики (GPO)
- Узнать доверительные отношения между доменами
- Запустить атаки типа Kerberoasting

Вся эта информация скрыта от неавторизованных пользователей.
Часто путь до Domain Admin начинается именно с такой "бесполезной" учетки.

## Поверхность атаки (какие сервисы цель)

В AD есть несколько сервисов, которые можно атаковать для получения первых учеток:

SMB (порт 445)
- Файловые шары, принтеры, удаленное администрирование
- Часто первая цель для password spraying

LDAP (порты 389/636)
- Протокол каталога для запросов к AD
- В misconfigured устройствах можно найти пароли

HTTP/HTTPS (порты 80/443)
- Внутренние порталы, SharePoint, device management
- Часто содержат пароли в логах, конфигах, коде

Kerberos (порт 88)
- Можно проверять существование пользователей (user enumeration)
- AS-REP Roasting для учеток без pre-authentication

DNS (порт 53)
- Помогает найти DC, mail серверы и другие ключевые машины

## Стартовые позиции (с чего начинаешь)

Unauthenticated (black-box)
- Есть доступ к сети, НЕТ учетных данных
- Нужно получить первый аккаунт через перебор, поиск паролей, coercion атаки

Authenticated (grey-box)
- УЖЕ ЕСТЬ низкопривилегированная учетка
- Сразу переходишь к перечислению и поиску путей повышения прав

## Ключевая мысль

В реальном пентесте самая сложная часть — часто именно получение первого аккаунта.
Дальше уже есть инструменты и методы.

# USER ENUMERATION ЧЕРЕЗ KERBEROS (без пароля)

Как проверить, существует ли пользователь в домене:

1. Отправляем AS-REQ запрос к KDC (порт 88)(pre-authentication)
2. Анализируем ответ:
   - KRB5KDC_ERR_PREAUTH_REQUIRED → пользователь СУЩЕСТВУЕТ
   - KRB5KDC_ERR_C_PRINCIPAL_UNKNOWN → пользователь НЕ СУЩЕСТВУЕТ

Это работает без знания пароля.

Инструмент: kerbrute
kerbrute userenum -d domain.local --dc IP_ADDRESS users.txt

Зачем это нужно:
- Составить список реальных пользователей для password spraying
- Найти слабые учетки
- Подготовить почву для AS-REP Roasting (если pre-auth отключена)
