# Domain - это группа пользователей и компьютеров под администрированием данного бизнеса
# Основная идея домена - администрирование компонентов сети в одном репозитории называемом Active Directory
# В Windows домене учетные записи хранятся в централизованном репозитории Active Directory(распределенная база данных)
# Сервер на котором работают необходимые службы AD(базы данных) называется Domain Controller(Контроллер домена)
### Например домен tryhackme.com НЕ ПУТАТЬ с DNS!!

# Основные преимущества конфигурирования домена:
* Централизованное управление идентификацией
* Настроить политику безопасности и права для пользователей и компьютеров

# Ядром любого windows домена является active directory domain service. Эта служба действует как каталог всех объектов, содержащихся в AD:
* users, groups, machines, printers, shares и др.

# Как работает аутентификация в домене:
Когда пользователь вводит пароль на любом компьютере домена, этот компьютер не проверяет пароль сам.
Он отправляет запрос к Domain Controller. DC сверяет данные со своей базой AD и отвечает:
"доступ разрешен" или "доступ запрещен".

# Почему ты не админ на школьном/рабочем компьютере:
Потому что в Active Directory настроены политики безопасности. Эти политики применяются 
ко всем компьютерам домена автоматически. Пользователям запрещено иметь локальные права 
администратора.

# Что дает домен (практически):
- Один раз создал пользователя в AD → он может войти на любой компьютер домена
- Один раз настроил политику → она применится ко всем компам
- Не нужно ходить к каждому компьютеру физически для настройки

# ОБЪЕКТЫ В ACTIVE DIRECTORY

AD хранит объекты. Основные типы:

## Users (Пользователи)
- Security principal (может быть аутентифицирован и получать права)
- Два типа:
  * People — обычные сотрудники
  * Services — учетки для запуска служб (например IIS, MSSQL)

### Пентест-нюанс про пользователей:
Служебные учетки (Service Accounts) часто имеют слабые пароли или встроены в скрипты.
Пример: в конфигах приложений можно найти пароль от учетки, у которой есть доступ к важным ресурсам.
Атакующий ищет такие учетки через SYSVOL, репозитории кода, память процессов.

## Machines (Компьютеры)
- Тоже security principal
- Создается когда компьютер вступает в домен
- Имя аккаунта = имя компа + $ (например DC01$)
- Пароль автоматический, 120 символов, ротируется
- Аккаунты компьютеров (ИМЯ$) — это легитимные учетки в AD. Они могут быть использованы атакующим для движения по сети, если получить их хэши из памяти.

### Пентест-нюанс про компьютеры:
Если атакующий получил хэш компьютера DC01$, он может выдать себя за контроллер домена.
Пример: атака Pass-the-Hash с хэшем компьютерного аккаунта позволяет получить доступ к другим серверам.

## Security Groups (Группы безопасности)
- Тоже security principal
- Можно добавлять пользователей, компьютеры, другие группы
- Нужны для выдачи прав на ресурсы (папки, принтеры и т.д.)

# ВАЖНЫЕ ГРУППЫ (ЗАПОМНИТЬ ДЛЯ ПЕНТЕСТА):

* Domain Admins:
Полный контроль над всем доменом.
Пример: если взломать любого пользователя из этой группы — весь домен твой.

* Server Operators:
Могут администрировать DC.
Пример: участник этой группы может залогиниться на контроллер домена и выполнить там команды.

* Backup Operators:
Может читать ЛЮБЫЕ файлы (включая AD базу) и писать ЛЮБЫЕ файлы.
Пример атаки: участник Backup Operators копирует файл C:\Windows\NTDS\ntds.dit с DC, затем извлекает оттуда хэши всех пользователей, включая админов.

* Account Operators:
Могут создавать и изменять учетные записи.
Пример эскалации: участник Account Operators создает нового пользователя, добавляет его в группу Domain Admins (если права позволяют) и получает полный контроль.

# ОРГАНИЗАЦИОННЫЕ ЕДИНИЦЫ (Organizational Units - OU)

- Контейнеры для группировки объектов
- Используются для ПРИМЕНЕНИЯ ПОЛИТИК (не для выдачи прав!)
- Один пользователь = одна OU (не может быть в двух)
- Обычно отражают структуру бизнеса (IT, Sales, HR...)

# OU vs Security Groups (ГЛАВНОЕ РАЗЛИЧИЕ)

OU применяется для политик (GPO). Security Group применяется для выдачи прав на ресурсы.
Пользователь находится только в одной OU, но может состоять во многих группах.
OU не дает прав доступа. Только Security Group дает права доступа.

Пример различия:
- OU "Бухгалтерия" используется, чтобы запретить бухгалтерам менять обои рабочего стола.
- Security Group "Finance_Access" используется, чтобы дать бухгалтерам доступ к папке \\DC\Finance.

# СТАНДАРТНЫЕ КОНТЕЙНЕРЫ (создаются автоматически):

- Builtin — стандартные группы Windows
- Computers — сюда попадают компьютеры при входе в домен (по умолчанию)
- Domain Controllers — все DC
- Users — стандартные пользователи и группы

### Пентест-нюанс про контейнеры:
Контейнер Computers часто игнорируют администраторы. Туда могут попасть компьютеры с устаревшими ОС или слабыми настройками. Пример: компьютер с Windows 7 в этом контейнере — легкая цель для атакующего.



##### ДЕЛЕГИРОВАНИЕ (Delegation) ####

Делегирование — это выдача конкретному пользователю или группе прав на выполнение определенных операций внутри OU, без назначения их в Domain Admins.

Пример из задания: Phillip (IT support) получил право сбрасывать пароли пользователям в OU Sales.

Как это делается: правой кнопкой на OU → Delegate Control → выбрать пользователя → выбрать какие права дать.

### Пентест-нюанс про делегирование:
Атакующий ищет неправильно настроенные делегирования. Например:
- Если пользователь может сбрасывать пароли админов (это баг)
- Если пользователь может добавлять себя в группу Domain Admins через делегирование
- Если делегированы слишком широкие права (например "Full Control" на OU)

### Опасные делегирования (пентест-чеклист)

Любое из этих делегирований на OU "Domain Controllers" или OU с администраторами — красный флаг:
- Full Control
- Reset Password
- Add/Remove members to group Domain Admins

Такое делегирование позволяет не-админу стать админом.

### Реальный пример атаки:
Атакующий находит, что пользователь "helpdesk_support" имеет право сброса паролей на OU "Admins". Он сбрасывает пароль администратора, логинится под ним и забирает весь домен.


#### ОРГАНИЗАЦИЯ КОМПЬЮТЕРОВ В OU (важно для пентеста) ####

По умолчанию все компьютеры (кроме DC) попадают в контейнер "Computers".

Это плохо с точки зрения безопасности и управления, потому что:
- Нельзя применять разные политики к разным типам устройств
- Все компьютеры в одной куче

# Рекомендуемое разделение компьютеров по OU (как минимум три категории):

1. Workstations (рабочие станции)
   - Компьютеры, за которыми работают обычные пользователи
   - На них НИКОГДА не должен заходить привилегированный пользователь (админ)
   
   Пентест-нюанс: если атакующий получил доступ к рабочей станции, он ищет там кэшированные пароли админов. Если админ когда-либо логинился на эту рабочую станцию — его хэш можно украсть.

2. Servers (серверы)
   - Машины, предоставляющие сервисы
   
3. Domain Controllers (контроллеры домена)
   - Самые чувствительные устройства
   - Хранят хэши паролей ВСЕХ пользователей

# Почему это важно для пентеста:

- Контроллер домена = главная цель (ntds.dit с хэшами всех учеток)
- Серверы = промежуточная цель (часто содержат ценные данные или учетки служб)
- Рабочие станции = входная точка (через пользователя можно двигаться дальше)

Если в домене все компьютеры валяются в одном контейнере "Computers" — это признак плохого администрирования и потенциальных проблем с безопасностью.
