## h2c Smuggling — объяснение атаки

### Что такое h2c?

Это способ установить HTTP/2 соединение **без шифрования**, через апгрейд HTTP/1.1.

Обычно HTTP/2 работает через TLS (h2). Но h2c позволяет сделать то же самое по незащищенному каналу.

### Как работает стандартный h2c upgrade:

1. Клиент отправляет HTTP/1.1 запрос:
   ```
   GET / HTTP/1.1
   Host: example.com
   Upgrade: h2c
   HTTP2-Settings: ...
   Connection: Upgrade
   ```

2. Сервер отвечает:
   ```
   HTTP/1.1 101 Switching Protocols
   Connection: Upgrade
   Upgrade: h2c
   ```

3. Дальше соединение переключается на HTTP/2 (бинарный протокол)


### Как это превращается в атаку:

Если между клиентом и сервером стоит **reverse proxy**:

[Клиент] → [Прокси (HAProxy)] → [Бэкенд]

Прокси **не всегда понимает h2c**. Он видит обычный HTTP/1.1 запрос с заголовком `Upgrade: h2c` и просто **проксирует его на бэкенд**.

**Ключевой момент:** После того как бэкенд ответил `101 Switching Protocols`, прокси **перестает анализировать трафик**, потому что думает, что соединение больше не HTTP, а какой-то другой протокол.

Теперь клиент и бэкенд общаются напрямую через HTTP/2, а прокси **просто туннелирует** данные, не проверяя их.

---

### Что это дает атакующему:

Прокси обычно блокирует доступ к `/private` (возвращает 403).  
Но если мы установим h2c-туннель через прокси, то:

1. Первый запрос (`/`) разрешен — апгрейд проходит
2. После апгрейда прокси больше не смотрит на содержимое
3. Мы отправляем HTTP/2 запрос на `/private` прямо в туннеле
4. Прокси пропускает его, потому что он в туннеле
5. Бэкенд отвечает

Мы **обходим ограничения прокси**.

### Визуально:

```
[Клиент]                    [Прокси]                  [Бэкенд]
    │                           │                         │
    │ HTTP/1.1 GET /            │                         │
    │ Upgrade: h2c              │                         │
    │──────────────────────────>│                         │
    │                           │ HTTP/1.1 GET /          │
    │                           │ Upgrade: h2c            │
    │                           │────────────────────────>│
    │                           │                         │
    │                           │ 101 Switching Protocols │
    │                           │<────────────────────────│
    │ 101 Switching Protocols   │                         │
    │<──────────────────────────│                         │
    │                           │                         │
    │ ====== HTTP/2 ТУННЕЛЬ (прокси не проверяет) ====== │
    │                           │                         │
    │ HTTP/2 GET /private       │                         │
    │───────────────────────────────────────────────────>│
    │                           │                         │
    │                           │ HTTP/2 GET /private    │
    │                           │ (прокси пропускает)    │
    │                           │                         │
    │ HTTP/2 Response (private) │                         │
    │<───────────────────────────────────────────────────│
```

---

### Почему это работает:

| Компонент | Что происходит |
|-----------|----------------|
| **Прокси** | Видит `Upgrade: h2c`, не понимает, проксирует на бэкенд |
| **Бэкенд** | Отвечает `101`, переключается на HTTP/2 |
| **Прокси** | Дальше просто туннелирует байты, не анализируя |
| **Клиент** | После апгрейда отправляет любой HTTP/2 запрос (даже запрещенный) |

---

### Ограничения:

- **Не влияет на других пользователей** (только твое соединение)
- Прокси должен **не поддерживать h2c сам** (иначе он перехватит апгрейд)
- Нужен TLS? Нет, h2c работает через HTTP (без шифрования)

---

### Инструмент h2csmuggler:

python3 h2csmuggler.py -x https://target.com:8200/ https://target.com:8200/private

- `-x` — URL, через который делаем апгрейд (должен быть разрешен прокси)
- Второй URL — целевой ресурс, который мы хотим получить (заблокирован прокси)
