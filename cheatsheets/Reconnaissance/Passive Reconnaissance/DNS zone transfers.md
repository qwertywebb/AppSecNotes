## Что такое DNS Zone Transfer

**DNS Zone Transfer (AXFR)** — это механизм, с помощью которого DNS-серверы обмениваются информацией друг с другом. Обычно это происходит между основным (master) и резервным (secondary) DNS-серверами, чтобы они имели одинаковую копию зоны.

**Проблема:** Если администратор забыл ограничить доступ в конфигурации dns-сервера allow-transfer { secondary_dns_ip; }(Это разрешит зонную передачу только авторизованным серверам.), любой человек может запросить полную копию зоны (все записи домена) и получить исчерпывающую информацию о структуре сети компании.

## Зачем это атакующему

Это один из самых лёгких и быстрых способов получить карту инфраструктуры. На основе информации, полученной при удачном Zone Transfer, злоумышленник может получить:
- IP-адреса серверов
- Имена внутренних поддоменов (включая staging, test, VPN, почтовые серверы)
- Контактные данные системных администраторов
- Используемые технологии

## Как выполнить Zone Transfer с помощью dnsenum

`dnsenum` — это утилита, которая автоматизирует процесс перебора и запросов DNS-записей.

### Базовый синтаксис:
```bash
dnsenum --enum zonetransfer.me
```

Эта команда автоматически найдёт DNS-серверы домена и попытается выполнить зонную передачу.

### С ручным указанием сервера:
```bash
dnsenum --dnsserver nsztm1.digi.ninja --enum zonetransfer.me
```

Здесь мы напрямую указываем, какой DNS-сервер опрашивать на предмет зонной передачи.

## Как выглядит результат

Если зонная передача разрешена, вы получите полный список DNS-записей домена. Ниже приведен фрагмент реального ответа от `zonetransfer.me`:

```
zonetransfer.me.        7200    IN      SOA     nsztm1.digi.ninja. robin.digi.ninja. 2014101601 172800 900 1209600 3600
zonetransfer.me.        300     IN      HINFO   "Casio fx-700G" "Windows XP"
zonetransfer.me.        301     IN      TXT     "google-site-verification=tyP28J7JAUHA9fw2sHXMgcCC0I6XBmmoVi04VlMewxA"
zonetransfer.me.        7200    IN      MX      0 ASPMX.L.GOOGLE.COM.
zonetransfer.me.        7200    IN      A       217.147.180.162
_sip._tcp.zonetransfer.me. 14000 IN     SRV     0 0 5060 www.zonetransfer.me.
vpn.zonetransfer.me.    4000    IN      A       174.36.59.154
owa.zonetransfer.me.    7200    IN      A       207.46.197.32
sqli.zonetransfer.me.   300     IN      TXT     "' or 1=1 --"
xss.zonetransfer.me.    300     IN      TXT     "'><script>alert('Boo')</script>"
```

## Разбор типов записей

| Тип записи | Что означает | Пример из вывода |
|------------|--------------|------------------|
| **SOA** (Start of Authority) | Содержит имя основного сервера и контакт администратора | `nsztm1.digi.ninja. robin.digi.ninja.` (здесь владелец: robin@digi.ninja) |
| **A** (Address) | Преобразует доменное имя в IPv4-адрес | `vpn.zonetransfer.me. → 174.36.59.154` |
| **MX** (Mail Exchange) | Указывает почтовые серверы | `ASPMX.L.GOOGLE.COM.` (компания использует Google Mail) |
| **TXT** (Text) | Произвольные текстовые данные | Часто содержат контакты, коды верификации или даже тестовые уязвимости (например, `sqli` и `xss`) |
| **SRV** (Service) | Указывает местоположение сервисов (протокол, порт, хост) | `_sip._tcp` — SIP-сервер на порту 5060 |
| **CNAME** (Canonical Name) | Создает псевдоним для домена | `staging` → `www.sydneyoperahouse.com` |
| **LOC** (Location) | Географические координаты | Координаты офиса компании |
| **PTR** (Pointer) | Обратная запись (IP → домен) | `164.180.147.217` → `www.zonetransfer.me` |
| **HINFO** (Host Information) | Информация об оборудовании и ОС | `"Casio fx-700G" "Windows XP"` |

## Что можно сделать с этой информацией

| Найденная запись | Что можно предпринять |
|------------------|-----------------------|
| `vpn.zonetransfer.me.` | Попытаться найти уязвимости в VPN-шлюзе |
| `owa.zonetransfer.me.` | Атака на почтовый сервер (брутфорс, подбор паролей) |
| `staging.zonetransfer.me.` | Тестовые серверы часто защищены хуже, чем боевые |
| `sqli.zonetransfer.me.` | Запись явно указывает на SQL-инъекцию — можно попробовать проэксплуатировать |
| `xss.zonetransfer.me.` | Тестовая XSS-уязвимость |
| Контакты в TXT-записях | Использовать для социальной инженерии или фишинга |
| MX-записи | Понять, используется ли Google Workspace, Microsoft 365 или свой почтовик |
| SRV-записи | Узнать, какие сервисы работают внутри сети |

## Практические команды

### 1. Ручная проверка (через dig)
```bash
dig axfr @nsztm1.digi.ninja zonetransfer.me
```

### 2. Автоматизированный перебор с dnsenum
```bash
dnsenum --enum zonetransfer.me
```

### 3. Проверка конкретного DNS-сервера
```bash
dnsenum --dnsserver nsztm1.digi.ninja --enum zonetransfer.me
```

### 4. Массовая проверка (если известен список DNS-серверов)
```bash
dnsenum --dnsserver 8.8.8.8 --enum example.com
```

### 5. Использование с прокси (через proxychains)
```bash
proxychains dnsenum --enum example.com
```

### 6. Парсинг результата для поиска конкретных поддоменов
```bash
dig axfr @nsztm1.digi.ninja zonetransfer.me | grep -E "\.zonetransfer\.me\." | awk '{print $1}' | sort -u
```

## Защита от Zone Transfer

Чтобы предотвратить такие утечки, администраторы должны:

- Ограничить зонную передачу только авторизованными вторичными серверами (используя `allow-transfer` в конфигурации BIND)
- Использовать TSIG-ключи для подписи транзакций
- Никогда не оставлять открытые зонные передачи на публичных DNS-серверах

## Вывод

DNS Zone Transfer — это мощный инструмент для разведки, который позволяет атакующему получить полную карту инфраструктуры без единого запроса к веб-серверу. Достаточно одного открытого DNS-сервера, чтобы узнать IP-адреса, имена внутренних поддоменов, используемые технологии и даже контакты администраторов. Проверка зонной передачи — одна из первых задач, которые выполняет пентестер при работе с целевым доменом.
