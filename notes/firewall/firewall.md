# Firewall & WAF — полная шпаргалка


## 🔥 Firewall (Межсетевой экран)

**Что это:** Стена между приложением и клиентом, которая по определенным алгоритмам отслеживает вредоносный трафик.

### Типы Firewall

| Тип | Принцип работы | Примеры |
|-----|----------------|---------|
| **Stateless** | Фильтрует пакеты на основе IP, port, protocol. Нет понятия сессий, потоков — каждый пакет как отдельный пользователь. | `iptables` без `conntrack`, ACL на старых Cisco |
| **Stateful** | Содержит таблицу соединений. Делает то же, что stateless, но помнит контекст сессии. | `iptables` + `conntrack`, `ufw` |
| **Application-Level Gateway (Proxy Firewall)** | Выступает в роли посредника, инициирует сеансы на уровне приложения. | `nginx` (reverse proxy), `squid`, SOCKS прокси |
| **Deep Packet Inspection (DPI)** | Работает с портами, соединениями, сканирует полезную нагрузку и ищет байтовые последовательности, характерные для подозрительного трафика. | `zapret`, `nDPI`, Cisco NGFW |
| **Next-Generation Firewall (NGFW)** | Проверяет всё: откуда трафик, кто отправитель, как зашифровано, можно ли безопасно расшифровать, должен ли пользователь получить доступ к этой услуге вообще. | Palo Alto Networks, Check Point, Kaspersky Next |

## 🌐 WAF (Web Application Firewall)

**Что это:** Специализированный firewall для веб-приложений. Парсит методы, хедеры, куки, query-параметры, body POST-запроса на предмет вредоносного payload'а.

### Три модели WAF

#### 1. Облачный (Cloud-based) Reverse Proxy
- **Как работает:** Ты направляешь DNS-записи своего сайта на Cloudflare или AWS. Весь трафик идёт через их сервера, где WAF анализирует его и только потом передаёт к тебе на сервер. WAF сам управляет TLS-сертификатами (терминирует HTTPS).
- **Примеры:** Cloudflare WAF, AWS WAF, Imperva.
- **Плюсы:** Не требует установки ПО на твой сервер, защищает от DDoS.
- **Минусы:** Трафик идёт через третью сторону, возможна задержка.

#### 2. Сетевой (Network Appliance / Physical)
- **Как работает:** Отдельное устройство (сервер или виртуальная машина) как шлюз перед серверами (обычно в дата-центре). Часто работает в паре с балансировщиком нагрузки (load balancer).
- **Примеры:** F5 Advanced WAF, Fortinet WAF.
- **Плюсы:** Высокая производительность, полный контроль над железом и конфигурацией.
- **Минусы:** Дорого, требует мощного администрирования.

#### 3. Встроенный в веб-сервер (Host-based / Embedded)
- **Как работает:** Модуль (например, ModSecurity) подключается прямо в твой веб-сервер — к nginx или Apache. Анализирует HTTP-трафик внутри самого процесса.
- **Примеры:** ModSecurity + OWASP Core Rule Set (CRS).
- **Плюсы:** Бесплатно, настраивается под твои нужды, не требует отдельного сервера.
- **Минусы:** Нагружает веб-сервер, требует опыта в настройке правил.

## ❌ От чего WAF **НЕ ЗАЩИТИТ**

| Уязвимость | Почему WAF бессилен |
|------------|---------------------|
| **DOM-based XSS** | Вредоносный код выполняется на клиенте (JavaScript), не отправляется на сервер → WAF его не видит. |
| **IDOR (Insecure Direct Object References)** | Требует понимания логики приложения: `../user/123/profile`. WAF не знает, что пользователь `123` не должен видеть профиль `124`. |
| **Новые payload'ы (0-day, обходные техники)** | Сигнатурный подход основан на известных шаблонах. Если атакующий модифицирует payload (например, кодирует, меняет регистр) — WAF может пропустить. |
| **Устаревшие правила** | Если сигнатуры не обновлять, WAF не узнает новые атаки (например, недавние уязвимости в WordPress-плагинах). |
| **Логические уязвимости** | Например, возможность зарегистрироваться с email'ом администратора, если нет подтверждения. WAF не проверяет бизнес-логику. |
| **Атаки на клиентскую часть (CSP, Cookie без флагов)** | WAF не контролирует, как браузер обрабатывает ответы. |
| **RCE через интерпретаторы (Log4Shell, SSTI)** | Требуют контекстного анализа; многие WAF пропускают, если payload закодирован или разбит. |

## 🛠️ Проверка наличия WAF / Firewall

### Инструмент: `wafw00f`
Автоматически проверяет заголовки, анализирует статус-коды при отправке разного payload'а.
```bash
wafw00f https://2ip.ru
```

### Nmap script
```bash
nmap -p 80,443 --script http-waf-fingerprint TARGET_IP
```

### Ручная проверка
```bash
# Отправить подозрительный payload и посмотреть ответ
curl -k "https://site.com/?id=1' OR '1'='1"
# Если вместо страницы с данными пришла ошибка 403 или редирект — возможно, WAF сработал.
```

## 🧠 Важно запомнить

| Тип защиты | Уровень | От каких атак защищает |
|------------|---------|------------------------|
| **Stateless Firewall** | L3-L4 | Только IP/порт/протокол |
| **Stateful Firewall** | L3-L4 | Контекст соединений |
| **WAF (сигнатурный)** | L7 | Известные веб-атаки (SQLi, XSS, LFI) |
| **WAF (поведенческий)** | L7 | Аномалии в трафике (но не логику) |
| **Безопасная разработка** | Приложение | IDOR, логические ошибки, бизнес-уязвимости |

**Вывод:** WAF — это не панацея, а часть эшелонированной защиты. Он не заменяет безопасную разработку, код-ревью, правильную настройку сессий и регулярное обновление правил. Для DOM-based XSS, IDOR и бизнес-уязвимостей нужно тестировать приложение вручную или с помощью DAST-сканеров.
